# ADR-001: Build a Chemistry-First Evidence-Governed Review Workbench

## Status

Accepted

## Date

2026-07-19

## Context

The repository demonstrates a deep asymmetric-allene case, provider and RAG
experiments, a staged dashboard, deterministic checks, and DOCX delivery. The
same history also created many phase-specific scripts, fixed Case 01 IDs and
counts, and several competing descriptions of the product.

The desired product is not a one-click model writer and does not require
multiple people to operate. One user should be able to provide a chemistry
question and papers/SI, review the work repeatedly at material decision points,
and receive one clean DOCX. Scientific facts must remain traceable to approved
evidence; disagreement and uncertainty must remain visible.

## Decision

Build a chemistry-first, case-neutral, evidence-governed review workbench with:

- one project and artifact contract shared by CLI, dashboard, Codex, and
  QoderWork;
- source records that separate `document_role`, `usage_role`, corpus
  governance, availability, and integrity, plus distinct parse artifacts;
- Evidence Discovery RAG for eligible candidate evidence and an authoritative
  Approved Claim Registry; any Approved Claim RAG/index is optional,
  rebuildable, and non-authoritative;
- immutable claim-version content, immutable claim-decision events, and a
  rebuildable registry projection; support and governance are projected rather
  than directly edited;
- immutable conflict versions, immutable conflict-decision events, and a
  rebuildable conflict projection; artifact errors remain outside scientific
  conflict authority;
- editable validated project configuration plus immutable resolved run,
  checkpoint, and release snapshots; scientific records and decisions remain
  immutable without event-sourcing ordinary configuration;
- independent Revision, Evidence support, Evidence governance, Artifact, and
  Checkpoint state domains;
- immutable RevisionEvent, ConflictRecord, CheckpointEvent, and ReleaseRecord
  histories;
- 15 logical checkpoints with human-required, human-on-exception, automatic,
  and not-applicable policies;
- a single-user repeated-review model with optimistic concurrency and visible
  invalidation;
- local-first checkpoint review and final DOCX export;
- explicit separation of `SELF_REVIEWED_DRAFT` and
  `EXPERT_REVIEWED_RELEASE`.

Case 01 remains a reference implementation and regression fixture. Its v5
revision is the golden calibration run, not the public schema. A later 20-40
paper chemistry project proves the large-review product path.

Frozen Phase 8 schemas retain their existing `source_role=MAIN|SI`. The Case 01
adapter maps that legacy field to generic `document_role` and never rewrites the
frozen schema or artifacts. Search-only candidates remain separate from source
records, and eligibility for retrieval or approved claims is derived from
explicit predicates rather than checkpoint passage alone.

One current registered version is allowed per claim lineage. A proposed version
does not supersede the current registered version until an evidence-registration
operation atomically registers the new version and supersedes the old one.
Unsupported unused candidates do not block the project; a blocking evidence
issue is derived only from an active intent or downstream dependency.

Academic controversy is not synonymous with source disagreement. It requires
current supported registered positions, structured comparability checks, and a
substantive remaining conflict. Conflict treatment is constrained by a
deterministic classification/status matrix, and any resolved synthesis is a
separately registered claim rather than prose generated by the conflict record.

## Alternatives considered

### Continue deepening Case 01 only

Rejected as the product strategy. It improves a single demonstration while
increasing hardcodes and does not prove reuse.

### Build a generic cross-domain AI Scientist

Rejected. Evidence semantics, ontology, evaluation, and claims of autonomy
would exceed demonstrated capabilities and weaken the chemistry product.

### Build an ordinary RAG plus one-click writer

Rejected. It cannot provide the accepted evidence-governance, conflict,
revision, and release guarantees.

### Require multiple human reviewers

Rejected as a product requirement. Multiple review moments are required, but
one user may complete them. External expert review is optional and raises the
release level only when bound to a specific DOCX hash.

### Rewrite the frontend with a large framework

Rejected for the current product slices. The existing Python and static-web
dashboard is sufficient for the first checkpoint UI and keeps Windows/QoderWork
portability simple.

### Add a database immediately

Rejected for the first product slice. Versioned contracts plus append-only
decision records and immutable run/snapshot/release closures remain inspectable,
portable, and adequate. A database may be reconsidered only after real scale or
concurrency evidence.

## Consequences

- Generic contracts and artifact identity must precede broad UI or case work.
- A correct source and a faulty derived parse are quarantined independently;
  status decisions require reason-coded events and validation references.
- Model output is advisory and cannot self-approve scientific evidence.
- Ordinary checkpoint approval cannot modify the registry. Reviewer synthesis
  depends only on eligible registered claims and an acyclic dependency graph;
  raw retrieval chunks cannot serve as its fact authority.
- Conflict blocking is dependency-local rather than project-global. A changed
  position invalidates only dependent controversy claims and downstream
  artifacts, followed by the required global consistency checks.
- ProjectManifest changes are detected against the immutable resolved config
  snapshot used by existing outputs. M0 reports `CONFIG_CHANGED` and affected
  stages instead of building manifest adoption, rollback, or projection
  infrastructure.
- ProjectManifest contains the offline capability ceiling but no provider
  profile. Any separately authorized model run records only non-sensitive actual
  provider/model identity and authorization evidence in RunManifest.
- M0 realizes contracts only through schemas, pure functions over explicit
  inputs, deterministic single-snapshot views, overwrite-refusing immutable
  outputs, adapters, fixtures, and offline tests. Target lifecycle, concurrency,
  dependency-propagation, and UI semantics remain design constraints for later
  separately authorized work.
- Offline-first language must distinguish optional authorized network/model
  execution.
- A user edit can invalidate downstream work, but only through material
  dependency edges; local rebuilds still receive global consistency checks.
- Historical Phase 8 assets remain evidence and fixtures rather than being
  generalized into the public ontology.
- Release records are immutable. A changed DOCX creates a new candidate instead
  of modifying a prior expert-reviewed release.
